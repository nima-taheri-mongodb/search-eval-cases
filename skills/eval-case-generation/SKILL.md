# Eval case generation

### Core Test Case Generation Configuration
Your task is to generate new, high-quality test cases for the 'mongodb-mcp-server' evaluation suite managed via Braintrust.

Review the background, design philosophy, and schema requirements below to construct valid test cases.

—

### 1. BACKGROUND & CONTEXT
- **Project Goal:** This project evaluates how accurately an LLM-powered assistant can translate natural language user requests into correct MongoDB Search and Vector Search queries and index management commands.
- **The Test Environment:** Tests run locally. A Docker container spins up a local MongoDB Atlas instance (on port 27017) pre-seeded with an isolated mock dataset ('movies') containing exactly 40 trimmed records to ensure rapid test execution and instant search embedding generation.
- **Evaluation Engine:** Braintrust orchestrates the run, using an 'LLM-as-a-Judge' to verify if the output generated by the assistant matches the ground-truth criteria.

---

### 2. CORE PHILOSOPHY (RULES OF ENGAGEMENT)
The llm_judge first checks whether the result is correct, then docks points based on how the assistant got there.
- **Check the result first.** When the query runs and returns meaningful results, grade on the final data state: does $response match the data, counts, or fields described in $reference_answer?
- **Inspect the query when the results don't prove correctness.** The returned documents sometimes can't tell you whether the assistant got it right: the query may not run in the test environment, or a different method could return the same documents. In those cases, grade the query the assistant submitted instead.
- **Reward the right method.** Lower the score when the assistant reaches the correct result the wrong way: skipping an available index, avoiding the expected native operator, or using an inefficient workaround.
- **Name the operators to check.** When method matters, write the specific stages, operators, or index names the judge should look for directly into the rubric. The reference answer stays the ground truth; the rubric just spells out which parts the judge must verify, so the check is a concrete checklist rather than an inferred comparison.
- **Inspect what was submitted.** To check the above, read the actual query the assistant sent to the tool in $conversation. A feature named only in prose does not count.
- **Ignore implementation details.** Do not require exact strings, spacing, or line breaks, and disregard markdown, commentary, and the specific MCP tool names used. Judge the database operation the assistant produced — its operators, stages, and structure — not the syntax or the tooling that delivered it.
- **Hybrid / fusion / rerank cases grade structure, not results.** `$rankFusion` / `$scoreFusion` / `$rerank` result tails vary with the unspecified `numCandidates`/`limit`, so for those cases skip result-set overlap entirely and grade the submitted pipeline's structure + topic per the [hybrid-search-rerank.md](hybrid-search-rerank.md) matrix.

### STRICT "EXPERT-VOICE" MANDATE
- All generated prompt inputs must be written in an **EXPERT VOICE**. This means they are written by a developer with knowledge of MongoDB features.
- **Style Rule:** Keep prompts simple, direct, and natural. Do NOT use overly verbose or dense technical jargon. Avoid artificial syntax complexity.
- **PROHIBITED (Non-Expert Use-Cases):** Do not write casual, vague business goals (e.g., "I am looking for search results that consider both what I typed in and the semantic meaning of my query").
- **REQUIRED:** Use straightforward technical intent.
- *Good, simple examples:* "Use hybrid search to find movies about...", "Perform a hybrid search for...", "Run a text search and bucket the output by decade."

---

### 3. TEST CASE SCHEMA REQUIREMENTS
Generate test cases adhering strictly to the following JSON structure matching our existing codebase:

```json
{
  "Category": "The top-level category (e.g., 'Text Search Query Construction' or 'Text Search Index Management')",
  "Subcategory": "The specific query/index feature sub-type (e.g., 'Post-Search Aggregation', 'Faceted Search', 'Index Creation')",
  "Name": "A short descriptive title for the test case",
  "input": {
    "db_seed": [
      // Define the target collection structure or dependencies needed for the sandbox setup
    ],
    "prompt": "The plain-English prompt given to the assistant under test."
  },
  "expected": {
    "reference_answer": "The exact stringified ground-truth MongoDB command text (e.g., 'aggregate([...])' or 'createSearchIndexes({...})').",
    "llm_judge": "Markdown string containing # Success criterion and an explicit # Scoring rubric awarding scales from 0 to 1.0 (e.g., 1.0 for perfect execution with index use, fractional points like 0.2/0.8 for partial correctness or lack of index use, 0 for failure)."
  },
  "metadata": {
    "category": "Matches top level Category",
    "subcategory": "Matches top level Subcategory",
    "name": "Matches Name",
    "description": "Short explanation of the specific feature test"
  }
}
```

#### Repo materialization (how the JSON above maps to this repo)
In this repo, cases are **YAML rows**, not JSON. Each row lives in
`dataset/<Dataset>/<Category>/<Subcategory>/cases.yaml` (taxonomy mirrors
`dataset/taxonomy.yaml`) and carries a stable `id`. `db_seed` is the seeded
`movies` collection plus its Atlas Search / Vector Search index definitions, and
`reference_answer` is a stringified `aggregate([...])` / `createSearchIndexes({...})`.

```yaml
- id: <uuid>            # added on first `pnpm push`/`pnpm pull`; omit for new stubs
  metadata:
    name: <Name>
    description: <Short explanation of the specific feature test>
    category: <Category>
    subcategory: <Subcategory>
  input:
    prompt: |
      On the 'movies' collection, ...        # natural-language only, expert voice
    db_seed:
      - movies:
          indexes:
            - definition: { mappings: { dynamic: true } }
              name: default
              type: search
            - definition:
                fields:
                  - { type: autoEmbed, path: plot, model: voyage-4-lite, modality: text }
              name: vector_auto_index
              type: vectorSearch
  expected:
    reference_answer: |
      aggregate([ ... ])
    llm_judge: |
      # Success criterion
      ...
      # Scoring rubric:
      1. If $response is empty, set score = 0.
      ...
```

Mirror an existing sibling row before inventing shape. For the exact YAML/db_seed
rules and validate/test/fix workflow see
[eval-case-authoring](../eval-case-authoring/SKILL.md); for verified terms,
indexes, and reference pipelines see
[mflix-search-queries](../mflix-search-queries/SKILL.md).
